# invoice-web 개발 로드맵

노션에서 입력한 견적서 데이터를 클라이언트가 고유 URL로 웹에서 확인하고 PDF로 다운로드할 수 있는 MVP 웹 애플리케이션

## 개요

invoice-web은 1인 개발자(관리자)와 견적서 수신 클라이언트를 위한 견적서 공유 시스템으로 다음 기능을 제공합니다:

- **노션 견적서 데이터 조회**: 노션 API를 통해 견적서 데이터를 실시간으로 읽어와 표시
- **견적서 웹 렌더링**: 구조화된 문서 형태(항목 테이블, 합계, 공급자/수신자 정보)로 견적서 표시
- **PDF 다운로드**: 견적서 웹 뷰를 PDF 파일로 변환하여 클라이언트가 보관 가능
- **고유 URL 기반 접근**: `/invoice/[id]` 형태의 고유 링크로 개별 견적서에 접근
- **견적서 목록 관리**: 관리자가 발행된 견적서 목록을 확인하고 공유 URL을 복사

## 개발 워크플로우

1. **작업 계획**

   - 기존 코드베이스를 학습하고 현재 상태를 파악
   - 새로운 작업을 포함하도록 `docs/ROADMAP.md` 업데이트
   - 우선순위 작업은 마지막 완료된 작업 다음에 삽입

2. **작업 생성**

   - 기존 코드베이스를 학습하고 현재 상태를 파악
   - `/tasks` 디렉토리에 새 작업 파일 생성
   - 명명 형식: `XXX-description.md` (예: `001-project-structure.md`)
   - 고수준 명세서, 관련 파일, 수락 기준, 구현 단계 포함
   - **API/비즈니스 로직 작업 시 "## 테스트 체크리스트" 섹션 필수 포함 (Playwright MCP 테스트 시나리오 작성)**
     - ⛔ **테스트 체크리스트 없는 API/비즈니스 로직 Task는 구현 단계 진행 불가**
     - Given-When-Then 형식으로 테스트 시나리오 작성 권장:
       ```
       - [ ] Given: [사전 조건] / When: [동작] / Then: [기대 결과]
       ```
     - 최소 포함 항목:
       - **정상 플로우**: 주요 성공 시나리오 (최소 1개)
       - **에러/예외 케이스**: 잘못된 입력, 인증 실패, 네트워크 오류 등 (최소 1개)
       - **경계값 테스트**: 빈 값, 최대 길이, 특수문자 등 (해당 시)
   - 예시를 위해 `/tasks` 디렉토리의 마지막 완료된 작업 참조. 예를 들어, 현재 작업이 `012`라면 `011`과 `010`을 예시로 참조.
   - 초기 상태의 샘플로 `000-sample.md` 참조

3. **작업 구현**

   - 작업 파일의 명세서를 따름
   - 기능과 기능성 구현
   - 각 단계 후 작업 파일 내 단계 진행 상황 업데이트
   - 각 단계 완료 후 중단하고 추가 지시를 기다림

   **🧪 테스트 게이트 (필수 — 구현 완료 후 반드시 수행)**

   > ⛔ **테스트 미통과 시 Task 완료 불가. 로드맵 업데이트 단계 진행 차단.**

   - 모든 API 엔드포인트 요청/응답 확인 테스트 (Playwright MCP)
   - 모든 사용자 인터랙션 E2E 테스트 (Playwright MCP)
   - 에러 핸들링 및 예외 상황 테스트
   - 테스트 결과를 작업 파일에 기록:
     - ✅ 통과 / ❌ 실패 (실패 사유 및 수정 내역 기록)
   - 실패 시: 구현 수정 → 재테스트 → 전체 통과 확인 후 진행

4. **로드맵 업데이트**

   - 로드맵에서 완료된 작업을 완료 표시로 변경
